**_TL;DR: With a few Fused UDFs you can turn any public docs site into a fully searchable RAG pipeline — connect data, process it, and expose it as an API._**

Retrieval-Augmented Generation (RAG) lets LLMs answer questions using real, up-to-date content instead of just their training data. The problem is that setting up a production-ready RAG pipeline usually means stitching together a scraper, a chunker, an embedding service, a vector store, and a query endpoint — often across multiple services and repos.

With Fused, the whole pipeline fits in a handful of Python UDFs. In this post we'll walk through how we built a semantic search system on top of the [Overture Maps documentation](https://docs.overturemaps.org/), and how you can reuse the exact same approach for any site.

## The 3-Step Pipeline

The architecture maps directly onto three questions:

1. **Connect** — How do you get the raw data?
2. **Process** — How do you turn it into searchable embeddings?
3. **Expose** — How do you let clients query it?

## Step 1: Connect — Crawl Any Docs Site into `llms.txt`

The first challenge is getting clean text out of a docs site. HTML is noisy: navbars, footers, sidebars, and scripts all get in the way. The [llms.txt standard](https://llmstxt.org/) solves this by defining a single, clean file that captures a site's full content in a format LLMs can actually use.

The `site_to_llmstxt` UDF accepts any docs site URL and a sitemap path, crawls every page in parallel, strips the boilerplate, and writes a single `llms.txt` to S3.

```python showLineNumbers
@fused.udf(cache_max_age='1d')
def udf(
    base_url: str = "https://docs.overturemaps.org/",
    site_name: str = "Overture_maps_documentation",
    mode: str = "full",          # "curated" (index) | "full" (page content)
    sitemap_path: str = "/sitemap.xml",
    max_pages: int = 0,          # 0 = no limit
    output_dir: str = "s3://fused-users/overture/demos/llms_txt/",
):
    """Generate an llms.txt from any public docs site with a sitemap."""
    ...
```

Internally the UDF:

1. Fetches and parses the sitemap (including sitemap indexes that reference sub-sitemaps)
2. Filters out non-content URLs (tag pages, assets, 404s)
3. Crawls every remaining page **in parallel** via `crawl_page.map()` with up to 10 workers
4. Extracts content with BeautifulSoup, converting headings, code blocks, and paragraphs into clean markdown
5. Writes the result to an S3 path like `s3://…/overture_maps_documentation/llms.txt`

Two modes are available:

| Mode | Output |
|------|--------|
| `curated` | One line per page — title, URL, short description |
| `full` | Full page text, lightly cleaned — best for RAG |

```python showLineNumbers
import fused

# Point at any docs site — the UDF handles the rest
result = fused.run(site_to_llmstxt, base_url="https://docs.overturemaps.org/")
print(result)  # s3://…/overture_maps_documentation/llms.txt
```

## Step 2: Process — Chunk, Embed, and Store in LanceDB

With the raw text in S3, the next step is turning it into a vector database. The `docs_ragging` UDF handles chunking, embedding, and writing to [LanceDB](https://lancedb.github.io/lancedb/) — a serverless vector store that runs directly on the mounted volume at `/mount/lancedb/`.

```python showLineNumbers
LANCEDB_TABLE_NAME = "overture_docs_17_04_2026"
LANCEDB_BASE_PATH = "/mount/lancedb/"
EMBEDDING_MODEL = "qwen/qwen3-embedding-8b"
CHUNK_SIZE = 1300       # characters per chunk
OVERLAP_PERCENT = 0.15  # 15% overlap between chunks

@fused.udf(cache_max_age=0)
def udf(
    table_name: str = LANCEDB_TABLE_NAME,
    overwrite: bool = True,
):
    import pandas as pd
    import requests

    ai = fused.load("ai_utils")

    response = requests.get(SOURCE_URL, timeout=60)
    content = response.text

    chunks = chunk_with_overlap(content, CHUNK_SIZE, OVERLAP_SIZE)

    df = pd.DataFrame({
        "chunk_id": list(range(1, len(chunks) + 1)),
        "code": chunks,
        "source_url": SOURCE_URL,
    })

    df = ai.embed_df(
        df,
        text_col="code",
        embedding_col="embedding",
        provider="qwen",
        embedding_model=EMBEDDING_MODEL,
        batch_size=400,
        max_workers=16,
    )

    ai.write_table(df, table_name, embedding_col="embedding",
                   base_path=LANCEDB_BASE_PATH, overwrite=overwrite)
```

### Chunking with overlap

Long documents need to be split before embedding. We use a fixed-size chunker with a 15% overlap so that context at chunk boundaries isn't lost:

```python showLineNumbers
def chunk_with_overlap(text, chunk_size, overlap_size):
    chunks = []
    start = 0
    while start < len(text):
        chunks.append(text[start : start + chunk_size])
        start += chunk_size - overlap_size
    return chunks
```

For the Overture docs (roughly 500K characters), this produces ~400 chunks at 1,300 characters each.

### Embeddings via ai_utils

Embedding is handled by `ai_utils`, a reusable UDF that wraps the [Qwen3-Embedding-8B](https://huggingface.co/Qwen/Qwen3-Embedding) model via OpenRouter. Batches are processed concurrently with up to 16 workers:


## Step 3: Expose — Semantic Search as a Fused API

Once the table exists, any client can query it. The `overture_docs_searching` UDF accepts a natural-language question, embeds it with the same model, and runs a cosine similarity search against the stored chunks:

```python showLineNumbers
@fused.udf
def udf(question: str = "Latest release?"):
    """Search Overture docs by natural language."""
    import pandas as pd

    ai = fused.load("ai_utils")
    collection_name = "overture_docs_17_04_2026"

    df = ai.search(question, collection_name, top_k=30)
    return df.head(5)
```

Under the hood, `ai.search` calls `embed_text` on the query and then runs a vector search against the LanceDB table:

```python showLineNumbers
def search(query, table_name, top_k=10, provider="qwen", base_path=DEFAULT_LANCEDB_PATH):
    query_vec = embed_text(query, provider=provider)
    results = _search_by_vector(table_name, query_vec, top_k=top_k)
    return results
```

Because this is a standard Fused UDF, it's immediately available as an HTTP endpoint. You can call it from Python, from Slack (via the [export API](/guide/data-input-outputs/export-api/slack)), from any notebook, or wire it into a larger agent loop.

```python showLineNumbers
import fused

results = fused.run(overture_docs_searching, question="How do I filter places by category?")
print(results["code"].iloc[0])
```

### Going further: Agentic RAG

The `ai_utils` library also includes a `run_agent` helper that wraps the search UDF into an OpenAI-compatible tool-calling loop. This lets an LLM decide *when* to search and *how* to synthesize the results into a final answer:

```python showLineNumbers
search_tool, search_handler = ai.make_search_tool(
    "overture_docs_17_04_2026",
    "Search Overture Maps documentation by natural language.",
)

result = ai.run_agent(
    query="What themes does Overture provide and which one has building footprints?",
    system_prompt="You are an expert on Overture Maps. Use the search tool to answer accurately.",
    tools=[search_tool],
    tool_handlers={"search": search_handler},
)
print(result["text"])
```

## Using This for Any Site

The pipeline is fully parameterized. To RAG a different docs site, change one line in `site_to_llmstxt` and one constant in `docs_ragging`:

```python showLineNumbers
# Step 1 — point at a different site
result = fused.run(site_to_llmstxt, base_url="https://docs.myproject.io/")

# Step 2 — pick a new table name and rerun
fused.run(docs_ragging, table_name="myproject_docs_v1", overwrite=True)

# Step 3 — query as before
fused.run(overture_docs_searching, question="How do I authenticate?")
```

## Conclusion

Three UDFs. No infrastructure. The same Fused execution model that lets you query spatial data tile-by-tile works just as well for building a full RAG pipeline:

- **Connect**: `site_to_llmstxt` crawls any docs site and writes clean `llms.txt` to S3
- **Process**: `docs_ragging` chunks, embeds, and stores in LanceDB on a shared volume
- **Expose**: `overture_docs_searching` is an instant HTTP endpoint any client can hit

Because every UDF is independently cacheable, versioned, and composable, you can swap in a different embedding model, switch to a different vector store, or add an LLM reranking step without rewriting the whole pipeline.
